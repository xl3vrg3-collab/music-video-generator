OVERVIEW
--------

LUMN Studio is a local-first AI film studio designed for producing short films,
trailers, and cinematic pieces with frame-accurate shot control, persistent
character identity, and motivated cinematic transitions.

It is NOT a music video generator, a slideshow maker, or a social-clip tool.
The system is built around a film production mental model: brief, preproduction,
shot design, anchor generation, video rendering, and editorial conform.

The V6 pipeline resolves the core failure mode of earlier AI video systems --
identity drift -- by separating visual identity from motion description. Reference
images carry the WHO and WHERE. Prompts carry only the HOW (camera, pose,
framing). This division is enforced at every layer of the stack.

Core engines:
  - Gemini 3.1 Flash (via fal.ai)    -- anchor still generation + image editing
  - Kling 3.0 V3/O3 (via fal.ai)    -- image-to-video, character element lock
  - Claude Sonnet 4.6                -- QA arbiter, candidate selection, audit
  - Claude Haiku 4.5                 -- fast evaluation, transition judge, critic
  - FFmpeg                           -- editorial conform, color grade, output

Runway is deprecated. All video production runs through fal.ai.

THE 7-STAGE WORKFLOW
--------------------

Stage 1:  BRIEF
  Define the project: genre, tone, world, characters, rough story beats.
  Sets the creative brief that drives all downstream decisions.

Stage 2:  DRAFTS & REFS
  Upload reference images: character photos, environment collages, costume refs,
  prop photos. These become the visual anchor of the production. No amount of
  text description can substitute for a well-chosen reference image.

Stage 3:  ASSETS (Preproduction)
  Generate structured reference sheets for all entities:
    - Character sheets: 8 panels (front, side, back, 3/4, face close-up x2,
      detail, emotional expression) -- photorealistic, no labels
    - Costume sheets: 8 panels (full-body, accessory details, fabric detail,
      layering view, movement study)
    - Environment sheets: 12 panels (establishing wide, interior POVs, ambient
      detail shots, lighting variants, time-of-day reads)
    - Prop sheets: detail panels for key story items

  All sheets generated by Gemini 3.1 Flash via fal.ai.
  Sheets are reviewed and one canonical "hero ref" is selected per entity before
  production can advance.

  Fast mode:       3-4 views per package.
  Production mode: 6-10 views per package.

Stage 4:  SCENES
  Script or import the beat sheet: a sequence of beats, each with characters
  present, location, emotion, and energy level. Scene planning assigns
  transitions between beats automatically using a transition map keyed on
  section flow (intro->verse, verse->chorus, etc.).

Stage 5:  SHOTS
  Each beat expands into individual shots with:
    - Shot type (extreme wide, wide, medium, close-up, etc.)
    - Camera angle (eye level, low angle, dutch, bird's eye, OTS, etc.)
    - Lens selection (24mm through 200mm)
    - Camera movement (push in, pull back, pan, crane, static, etc.)
    - Duration (3-15s)
    - Kling tier (v3_standard, v3_pro, o3_standard, o3_pro)
    - Character elements (which preproduction packages to bind)

  Shot priority is set per-shot: standard, hero, or insert.
  Hero shots receive Sonnet QA. Standard shots run Haiku evaluation.

Stage 6:  RENDER (V6 Pipeline)
  The full pipeline runs in phases, each independently invokable:

    anchors  ->  select  ->  review  ->  clips  ->  conform

  See V6 Pipeline section below for full detail.

Stage 7:  OUTPUT
  Conformed MP4 at the target aspect ratio (16:9, 9:16, 1:1, 4:5).
  Optional color grade applied via FFmpeg (none/warm/cold/vintage/
  high_contrast/noir/cyberpunk/sepia).
  Optional watermark, credits, thumbnail extraction.
  Export presets for platform delivery (YouTube, Instagram, TikTok).


V6 PIPELINE: GEMINI ANCHORS -> SONNET QA -> KLING VIDEO -> FFMPEG CONFORM
--------------------------------------------------------------------------

Entry point:   scripts/pipeline_v6_gemini_kling.py
Web UI entry:  /api/v6/* endpoints in server.py
Config:        output/pipeline/production_plan_v4.json
Packages:      output/preproduction/packages.json

  Phase 1 -- ANCHORS (Gemini 3.1 Flash)
  ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
  For each shot in the production plan:
    1. Resolve character, costume, and environment packages assigned to the shot
    2. Select the approved hero ref images for each package
    3. Construct a lean anchor prompt -- camera/framing/pose ONLY
       (refs carry identity; prompts must NOT re-describe what refs already show)
    4. Call gemini_edit_image() with up to ~10 reference images
       Endpoint: fal-ai/gemini-3.1-flash-image-preview/edit
    5. Download resulting PNG to output/pipeline/anchors_v6/

  Optionally run multiple candidates (3x) per shot before selection.
  Anchor stills are stored with metadata binding them to their shot.

  Phase 2 -- SELECT (Sonnet QA)
  ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
  When 3 candidates exist per shot, Sonnet picks the best one:
    - Vision analysis of all candidate images
    - Scores on identity fidelity, pose accuracy, environment match, composition
    - Returns selection with reasoning
  Selected anchor is marked in the production plan and used downstream.

  Phase 3 -- REVIEW (Sonnet Transition Audit)
  ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
  Sonnet reviews adjacent anchor pairs across the full sequence:
    - Checks spatial continuity (same location, consistent lighting,
      180-degree rule compliance)
    - Flags identity drift between shots of the same character
    - Recommends transition type for each cut based on shot size jump
    - Flags shots that need a re-establishing wide before proceeding
  Results saved to: output/pipeline/learning/sonnet_review_v6.json

  Phase 4 -- CLIPS (Kling 3.0 via fal.ai)
  ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
  For each approved anchor:
    1. Build the Kling motion prompt: movement description only (15-40 words)
       No subject re-description. No sound words. No color commentary.
    2. Optionally attach character elements for Kling's native consistency system:
         elements = [{"frontal_image_url": "...", "reference_image_urls": ["..."]}]
       V3 endpoints support up to 4 elements. O3 does not use elements.
    3. Optionally set end_image_path for transition control between two anchors
    4. Submit to Kling via fal.ai queue and poll for result
    5. Download MP4 to output/pipeline/clips_v6/

  Kling tier selection:
    v3_standard   $0.084/s   default, solid quality, reasonable queue time
    v3_pro        $0.112/s   higher quality, slower queue
    o3_standard   $0.084/s   newer architecture, no elements support
    o3_pro        $0.392/s   premium quality, longest queue

  Phase 5 -- CONFORM (FFmpeg)
  ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
  Editorial conform via lib/video_stitcher.py and lib/editorial_conform.py:
    1. Normalize all clips to target resolution (default: 1920x1080)
    2. Apply per-shot color grade preset
    3. Concatenate with per-shot transition type
    4. Apply audio track (optional) with fade in/out
    5. Export final MP4 to output/pipeline/final/

  Supported transition types (19 total):
    hard_cut, crossfade, dissolve, fade_black, wipe_left, wipe_right,
    slide_left, slide_right, slide_up, slide_down, zoom_in, glitch,
    rotate, blur_transition, flash_white, flash_black, pixelate,
    squeeze, circular_reveal

  Transition type is assigned per cut from the Transition Intelligence system,
  not defaulted uniformly across the production.


PREPRODUCTION SYSTEM
--------------------

Module: lib/preproduction_assets.py
API:    /api/preproduction/*, /api/pos/*
Data:   output/preproduction/, output/prompt_os/

The preproduction system sits between story planning and shot generation. Its
purpose is to replace text-only character/environment descriptions with approved
visual reference packages that drive all image and video generation.

Package types:
  character      Visual identity package -- 8-panel photorealistic reference sheet
  costume        Wardrobe package -- clothing, materials, accessories
  environment    Location package -- establishing wides + interior details + light reads
  prop           Key story objects -- multi-angle detail sheet

Package status flow:
  draft -> generating -> generated -> approved / rejected / missing

A package cannot be used in production until its status is "approved". This gate
prevents shots from generating against incomplete or rejected references.

Sheet generation:
  All sheets are generated by Gemini 3.1 Flash (text-to-image mode, no refs needed
  at the sheet stage). The system composes structured prompts using HERO_PROMPTS
  templates that enforce multi-panel layouts, white backgrounds, consistent
  character identity across panels, and zero text/labels in the output image.

Hero ref selection:
  After a sheet is generated, the user selects one canonical "hero ref" image per
  package. This single image becomes the primary reference for all downstream
  anchor generation and Kling element binding for that entity.


PROMPT OS (PROMPT OPERATING SYSTEM)
-------------------------------------

Module: lib/prompt_os.py
Data:   output/prompt_os/
API:    /api/pos/*

The Prompt OS is the central registry for all creative entities in a production.
It stores structured records -- not just text prompts -- combining visual refs,
descriptive metadata, and assembly rules.

Entity types:

  Characters    Unique people or beings. Each record stores:
                  - Name, description, distinctive features
                  - Attached photo/sheet path (the hero ref)
                  - Default framing preference
                  - Identity lock notes

  Costumes      Wardrobe packages linked to one or more characters.
                  - Full-body and detail ref images
                  - Material, color, period notes
                  - Associated props and accessories

  Environments  Locations. Each record stores:
                  - Canonical wide reference image
                  - Interior/exterior designation
                  - Lighting notes (golden hour, overcast, artificial, etc.)
                  - Time-of-day variants

  Props         Key story objects.
                  - Multi-angle detail photos
                  - Scale reference notes
                  - Which characters handle this prop

  Scenes        Assembled compositions: character + costume + environment + props.
                  - Emotional tone
                  - Energy level
                  - Style lock notes

  Style Locks   Production-wide visual rules:
                  - Color palette constraints
                  - Film stock or look reference
                  - Camera movement restrictions
                  - Forbidden visual elements

  World Rules   Narrative and physical rules:
                  - Diegetic constraints (no phones in period pieces, etc.)
                  - Character knowledge/motivation notes

Prompt assembly (lib/prompt_assembler.py) pulls from these records when building
anchor and motion prompts, ensuring every shot inherits the correct visual context
without the human operator having to manually re-state world details per shot.

Data files in output/prompt_os/:
  master_prompts.json     All prompt entities
  characters.json         Character records
  costumes.json           Costume records
  environments.json       Environment records
  scenes.json             Scene composition records
  style_locks.json        Production-wide style rules
  world_rules.json        Narrative/physical rules
  continuity_rules.json   Continuity enforcement rules
  props.json              Prop records
  project_style.json      Top-level project style definition
  sheets/                 Generated reference sheet images


TRANSITION INTELLIGENCE SYSTEM
--------------------------------

Modules: lib/transition_judge.py, lib/transition_strategy.py,
         lib/prompt_packs/ (14 packs)

The Transition Intelligence system prevents jarring cuts by analyzing adjacent
shot pairs and selecting the appropriate transition strategy before rendering.

5+1 Dimension Scoring (transition_judge.py):

  Dimension      Weight   What it measures
  ---------      ------   ----------------
  Identity         3.0    Subject looks consistent across the cut
  Pose             2.0    Body position and angle compatible
  Camera           1.5    Framing, height, and lens compatible
  Scene            1.5    Same environment, lighting, time of day
  Motion           2.0    Implied motion flows logically
  Overall          --     Weighted composite of the above

Scores run 0.0 (broken) to 1.0 (seamless). The composite score drives strategy
selection in transition_strategy.py.

Two scoring modes:
  Heuristic    Fast structural analysis from shot metadata (free, no API call)
  Vision       Actual image analysis via Claude Haiku or Sonnet (API cost applies)

Strategy Engine (transition_strategy.py):

  Strategy              Cost Tier    Description
  --------              ---------    -----------
  motivated_cut         free         Hard cut driven by narrative (action, eyeline,
                                     smash, match, impact, audio-led)
  direct_animate        standard     Animate straight from A to B anchor
  end_variants          premium      Generate end-frame variants of clip A
  bridge_frame          premium      Generate intermediate still bridging A to B
  regenerate_pair       hero         Regenerate both clips with matched entry/exit

  Fallback chain:
    keep_start -> regen_end_variants -> bridge_frame -> motivated_cut -> regen_pair

  Cost tier (standard/premium/hero) is set per shot and controls which strategies
  are available. Hero shots get the full fallback budget; standard shots use
  motivated_cut and direct_animate only.

Motivated cut types (11 variants):
  motivated_cut, hidden_cut, whip_pan_cut, object_wipe_cut,
  foreground_wipe_cut, blink_cut, reaction_cut, insert_cut,
  match_cut, impact_cut, audio_led_cut

Self-Healing Learning (lib/learning_system.py):
  Every shot attempt is logged with its inputs, scores, and failure types.
  After a batch, Haiku clusters failure patterns and Sonnet proposes rule updates.
  Active rules are written to output/pipeline/learning/prompt_rules.json and
  applied to future shots in the same production.

  Storage files in output/pipeline/learning/:
    shot_attempts.json      Per-shot attempt log
    failure_clusters.json   Grouped failure patterns
    prompt_rules.json       Active prompt rule adjustments
    thresholds.json         Current scoring thresholds
    optimizer_history.json  History of optimizer runs
    sonnet_review_v6.json   Sonnet full-sequence transition audit


SPATIAL CONTINUITY SYSTEM
--------------------------

Rules enforced by the Sonnet review phase and the Prompt OS world rules:

  Canonical Wides         Every location must have one approved wide establishing
                          shot. This wide is re-used (not re-generated) as the
                          spatial anchor for that location. Shot generation pulls
                          from it.

  180-Degree Rule         The axis of action is locked per scene. Camera positions
                          are constrained to the approved side of the line.

  Light Locks             Approved lighting direction and quality for each
                          environment are stored in the environment package.
                          Anchor prompts reference the locked light.

  Re-establishing Shots   After a significant temporal or spatial break, the
                          Sonnet review flags the need for a re-establishing
                          wide before close or medium shots resume.

  Identity Gate           The first 3-8 seconds of any character's appearance in
                          a scene must be a medium or close shot that locks their
                          identity before wider or faster shots proceed. This
                          prevents the audience from losing track of who is on screen.


CLAUDE QA SYSTEM (HAIKU / SONNET)
-----------------------------------

Module: lib/claude_client.py

Two-tier evaluation designed to minimize cost while preserving quality gates.

  Model            ID                              Primary Use
  -----            --                              -----------
  Haiku 4.5        claude-haiku-4-5-20251001       Default critic for all shots
  Sonnet 4.6       claude-sonnet-4-6               Hero shots, borderline cases,
                                                    escalations, meta-analysis

Escalation triggers (automatic):
  - Shot priority is "hero"
  - Haiku overall score falls in borderline range (0.45 to 0.65)
  - Two or more failure types detected in one shot
  - Two or more retry attempts already made on that shot
  - Explicit force_escalate flag set

All calls return structured JSON. Vision calls include base64-encoded images
inline. The client handles markdown fence stripping and fallback JSON extraction.

QA call types:
  judge_transition_vision()     Haiku-first transition scoring with auto-escalation
  critique_render_vision()      Post-render image/video quality critique
  call_json() / call_text()     General structured and unstructured reasoning tasks

USAGE: CLI WORKFLOW
--------------------

1. Define your production plan JSON or use the web UI to build it.
2. Complete preproduction: generate and approve all entity packages.
3. Run anchors phase -- generates one still per shot via Gemini.
4. Run select phase (if using 3x candidates) -- Sonnet picks the best.
5. Run review phase -- Sonnet audits the full anchor sequence.
6. Address any flags from the review (re-run individual anchors as needed).
7. Run clips phase -- Kling generates video from each selected anchor.
8. Run conform phase -- FFmpeg stitches the final output.

To regenerate a single shot anchor without re-running the full batch:
  Edit that shot entry in production_plan_v4.json to clear anchor_path, then
  re-run the anchors phase. It will skip shots that already have anchors.

To re-render a single clip:
  Clear clip_path for that shot entry and re-run the clips phase.


USAGE: WEB UI WORKFLOW
-----------------------

1. Open http://localhost:3849.
2. Navigate to Prompt OS. Create character, costume, and environment entities.
   Upload reference photos for each entity.
3. In Preproduction, generate reference sheets for each entity. Review and approve.
4. In Scene/Beat Editor, build the beat sheet. Assign characters and locations.
5. In Shots Workspace, configure per-shot camera specs. Set shot priorities.
6. In V6 Pipeline Panel, run: anchors -> select -> review -> clips -> conform.
7. In Timeline, review clips. Adjust transition types per cut if needed.
   Re-render individual clips directly from the timeline interface.
8. In Output, download the final MP4. Export for platform if needed.

BRAND + UI AESTHETIC
--------------------

LUMN's interface is cinematic and artistic, not SaaS. The UI is meant to feel
like a filmmaker's production wall, not a dashboard.

  Typography        Inter Tight, light weight 300 for body copy.
                    Wide letter-spacing (6-8px) uppercase for labels.
                    No monospace accent fonts in UI chrome.

  Surfaces          Muted rgba whites on dark backgrounds.
                    12px backdrop-blur translucent panels.
                    No glowing drop shadows. No neon glows.

  Welcome imagery   brand_assets/LUMN_Welcome_Dark_*.png and
                    brand_assets/LUMN_Welcome_White_*.png
                    (bear_dark / bear_light atmospheric anchors). The welcome
                    page is the heaviest visual surface in the app -- all other
                    panels are quiet.

  Retired palette   Earlier iterations used a cyberpunk neon look
                    (cyan #00E5FF, magenta #FF5E00, amber, violet, glowing
                    shadows, JetBrains Mono). That aesthetic is retired.
                    Any reference to it in older CSS, comments, or docs
                    should be treated as historical. The "cyberpunk" color
                    grade preset in video_stitcher.py is a post-production
                    color look for output video, NOT the UI.

  Voice             Cinematic, film-production-binder feel. Not dashboard.
                    Labels are short. Status indicators are subtle. The user
                    should feel they are directing a production, not operating
                    a SaaS tool.


KEY DESIGN DECISIONS
---------------------

1. REFS NOT TEXT
   Visual identity is carried entirely by reference images, not prompt text.
   Anchor prompts describe only camera, pose, and framing. Describing the
   subject in a prompt when a reference image is present dilutes the reference
   signal and causes drift. This is the single most important rule in the system.

2. NO ENGINE MIXING
   A production uses exactly one video engine from start to finish. Currently
   that is Kling 3.0 via fal.ai for all video. If a clip fails QA, it is
   re-rendered with the same engine using a different prompt strategy -- it is
   never silently substituted with a different engine.

3. NO MULTI-SHOT API CALLS FOR CONTINUITY
   Kling's multi-shot API mode causes character drift between sub-shots. Every
   shot in LUMN is generated as an independent call with a single anchor image.
   Per-shot QA is applied before advancing to the next shot.

4. IDENTITY GATE
   The first appearance of any character in a scene must be a medium or close
   shot. This locks the audience's read of that character's identity before
   wider or faster shots that compress the identity signal.

5. MOTIVATED CUTS ONLY
   Generic dissolves or crossfades are not applied by default. Every transition
   is evaluated by the Transition Intelligence system and assigned a type with
   a narrative justification (action cut, eyeline cut, smash cut, match cut,
   etc.). A dissolve is only used when it is the right choice for the specific
   emotional context of that cut.

6. NO VISIBLE WINDOWS (WINDOWS PLATFORM)
   All subprocess calls on Windows use windowsHide=True / CREATE_NO_WINDOW.
   FFmpeg and all other subprocesses run invisibly in the background.

7. QA BEFORE ADVANCING
   Generated images are vision-analyzed by Claude before the pipeline advances
   to the next stage. No shot proceeds to video rendering if its anchor fails
   the QA gate. The system re-generates or escalates rather than silently passing
   failing assets downstream.


FUTURE ENHANCEMENTS
--------------------

  Multi-character element binding
    Bind up to 4 character packages simultaneously for ensemble shots.
    Currently limited by Kling's elements system maximum of 4 total elements.

  Temporal extension via clip stitching
    Stitch two 5s clips from the same anchor to produce 10s compound shots
    without incurring O3 Pro pricing for every long shot.

  Audio-driven shot timing
    Use audio analysis data (beat/section from audio_analyzer.py) to suggest
    shot durations that align cuts to musical or narrative rhythm points.

  Veo 3.1 integration
    fal.ai is expected to carry Veo 3.1 when API access opens. Architecture
    supports adding new engines via fal_client.py without changing upstream
    pipeline logic.

  Shot coverage intelligence (lib/coverage_system.py)
    Automatic detection of coverage gaps: scenes missing establishing wides,
    characters without identity-locking medium shots, dialogue scenes without
    over-the-shoulder setups.

  Taste profile learning (lib/taste_profile.py)
    Per-user preference model that adjusts default tier selection, transition
    type preferences, and color grade choices based on approval/rejection history.

  Project import from screenplay PDF
    Parse standard screenplay format (Fountain or PDF) to auto-generate beat
    sheets and shot lists, with manual review before production starts.

  Collaborative review mode
    Export a production package (anchor stills + shot data) to a shareable
    review link. Collaborators can approve/reject shots and add notes without
    requiring local server access.


NOTES ON DEPRECATED ENGINES
-----------------------------

Runway
  Deprecated. All Runway API references in video_generator.py are legacy stubs
  retained for backward compatibility with older project files. No new production
  should use Runway endpoints. The RUNWAY_COST_PER_SEC constant and related
  polling code remain in video_generator.py but are not called by the V6 pipeline.

Luma Dream Machine (Ray2)
  Legacy engine, not recommended for current productions. Supported in
  video_generator.py for experimental use but not part of the V6 workflow.

Grok (xAI)
  Legacy text-to-video engine. Retained for experimental use. Fixed 8-second
  duration. Not part of the V6 pipeline.

OpenAI image + Ken Burns
  Legacy still-to-video fallback via GPT image generation plus FFmpeg Ken Burns
  animation. Available but not recommended for production work.
